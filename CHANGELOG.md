# Changelog

Weebill is a fork of [sylph](https://github.com/bluenote-1577/sylph); this file records
changes made in weebill. The upstream sylph release history is in
[CHANGELOG-sylph.md](CHANGELOG-sylph.md).

## Unreleased

### Added

- Added `sketch --tolerate-empty-inputs`, which treats a read input containing zero reads (an empty file or stream — e.g. a FIFO fed by an SRA that has no unpaired reads) as a valid zero-read sketch instead of a failure. This is aimed at `--merge`: an empty single-end/paired-end/interleaved stream no longer aborts the merge (previously it was reported as *"not a valid fasta/fastq file"* and, under `--merge`, refused as a partial sketch). Genuine errors are still errors — only one mate of a pair being empty remains a failure — and a run where *every* read input is empty still fails.
- `sketch` now errors out when read inputs were given but every read stream (paired, interleaved, and single-end) turned out empty, instead of silently exiting 0 having sketched nothing. This matters when reads are streamed in through FIFOs and an upstream producer emits nothing — the run now fails loudly rather than producing no output (or an empty merged sketch). A single non-empty stream among empty ones is still fine; only a zero total across all streams is an error.
- **AVX-512 k-mer extraction.** Added an AVX-512 path (`avx512_seeding.rs`) for the FracMinHash marker-extraction hot loop that drives all read/genome sketching. Where the existing AVX2 path rolls, hashes and thresholds four k-mers per iteration (four 64-bit lanes of a `__m256i`), the new path does eight at a time (a `__m512i`) and compacts the k-mers that pass the sampling threshold to the output with a single `vpcompressq` (`_mm512_mask_compressstoreu_epi64`) instead of a per-lane scalar branch. The kernel is selected at runtime (`avx512f` → AVX-512, else `avx2` → AVX2, else scalar); on hardware without AVX-512 nothing changes. Output is **bit-for-bit compatible**: the AVX-512 path emits exactly the same multiset of hashes as the AVX2 path for every input (verified by an exhaustive length/k/c fuzz), so sketches are identical regardless of which CPU built them. On an E. coli benchmark (~44 Mbp, k=31) marker extraction is **~1.6× faster than AVX2** at the default `c=200` and up to **~2.0× faster** at the dense subsampling rates used by two-stage screening (`c=3`), i.e. ~2.1× over scalar. The idea (widest-vector parallel k-mer processing + branchless survivor compaction) is from Igor Martayan's PhD thesis on SIMD k-mer algorithms (https://github.com/imartayan/phd, "simd-minimizers"). A `cargo run --release --example bench_seeding` harness reproduces the correctness check and timings.
- Added reference-based delta compression of sample sketches (`weebill ref-build` + `weebill ref-compress`). `ref-build` turns genome database sketches into a k-mer dereplicated reference (`.sylref`): a two-level (species representative / strain) scheme assigns each k-mer to at most one genome, preferring representatives (robust to strains contaminated with another species' sequence), with strains of a species placed contiguously and conserved k-mers kept in a shared pool. `ref-compress` then encodes a sample (`.sylsp` → `.sylspr`) by storing, for each detected genome, *which* of its reference hashes are present (adaptive bitmask / present-Rice / absent-Rice, plus delta-coded genome ids), with only error/novel hashes stored explicitly; `--decompress` reconstructs the exact `.sylsp`. On a mixed E. coli sample this was ~60% smaller than the `.sylspc` format and ~80% smaller than bincode, losslessly (profile/query output is identical after a round trip). `query` and `profile` can also read `.sylspr` samples directly with `--reference <ref.sylref>`, decoding them on the fly without first running `ref-compress --decompress`; output is identical to running on the original `.sylsp`. Decoding always requires the same reference DB used to compress. The `.sylref` is a **two-stage seekable database** so large references don't have to be loaded in full: stage 1 is a small, uncompressed sparse index (1/N of each genome's distinctive k-mers, controlled by `ref-build --sparse-subsample`) that is queried to find the genomes a sample contains; stage 2 then decodes only those genomes' dense Golomb–Rice blocks on demand (cached across samples), with the shared pool loaded once. Missing a genome at stage 1 only costs ratio (its k-mers fall back to novel coding), never correctness — decompression loads exactly the genome blocks a sample records. `ref-build` itself is **streaming and parallel**: it reads compressed databases one genome at a time (never holding them all in RAM), routes k-mers into hash-partitioned on-disk scratch shards, then builds ownership for the shards in parallel — so peak RAM is bounded by `--max-ram <GB>` (a soft target that sizes the partition count) rather than by the whole database. `--tmp-dir` controls scratch location (needs roughly the input size of free space). The result is byte-identical regardless of the partition count.
- Added `--compressed-output` (analogous to `-o`) and `--compressed-database` (analogous to `-d`) to `weebill sketch`. These write compressed sketches (`.syldbc` databases / `.sylspc` samples) that are substantially smaller on disk (~30%+ smaller databases, ~55% smaller samples on test data). The FracMinHash values are sorted, delta-encoded and Golomb–Rice coded — near-optimal for the geometric gap distribution of uniform hashes — with hashes stored separately from their multiplicities, and the whole payload is then wrapped in a fast zstd frame that mops up the repetitive metadata (genome/contig names) and count stream. Rice reaches the entropy of the hash set so writing stays nearly as fast as the uncompressed format, and zstd adds negligible time. `query`, `profile`, and `inspect` transparently read both the legacy and compressed formats (detected by content, not extension), so the two can be mixed freely.
